# Portfolio Language Standard

This note defines the writing standard for portfolio case studies across `III.IV`.

The goal is not to make the writing casual.

The goal is to make it sound like a real analyst wrote it:

- clear
- commercially grounded
- specific
- restrained
- human

## Core writing rule

If the analysis is strong but the language sounds templated, the work still loses credibility.

Strong writing should feel authored, not generated.

## Tone

Use a calm analyst voice.

The writing should sound like:

- someone who understands the business question
- someone who can make a judgment
- someone who is willing to be specific
- someone who knows the limits of the evidence

Do not write like:

- a consultant trying to sound universally correct
- a dashboard summary generator
- a polished but generic strategy memo

## Style principles

### 1. Prefer concrete language

Use direct nouns and verbs.

Prefer:

- `late delivery`
- `basket size`
- `repeat purchase`
- `margin pressure`
- `stockout risk`

Over:

- `commercial performance dynamics`
- `value realisation`
- `operational complexity signals`

### 2. Keep claims tight

If the data only supports a modest point, write a modest point.

Prefer:

- `This suggests...`
- `What stands out is...`
- `The clearest pattern is...`
- `The data points to...`

Over:

- `This clearly demonstrates...`
- `The analysis proves...`
- `This surfaces the most important signal...`

### 3. Reduce repeated portfolio phrases

Avoid overusing phrases such as:

- `decision-ready`
- `commercially grounded`
- `the case shows`
- `the work focuses on`
- `the analysis highlights`
- `growth opportunity`
- `strongest`
- `surface the most important`

These are not banned, but they should be used sparingly.

### 4. Vary rhythm naturally

Do not make every sentence the same length or structure.

Use a mix of:

- short decisive lines
- medium explanatory lines
- occasional longer sentences where the logic needs it

### 5. Write recommendations like a person making a call

Recommendations should sound owned and practical.

Prefer:

- `Prioritise this first.`
- `Do not treat that as a clean win without checking returns.`
- `Use this channel for reach, not for margin quality.`

Over:

- `This may present an opportunity to optimise strategic execution.`

### 6. Let uncertainty stay visible

If the project uses synthetic data or rule-based scoring, say so plainly.

Prefer:

- `This is a synthetic case study designed to test the decision logic.`
- `These action labels are triage cues, not causal proof.`

Over:

- smoothing weak evidence into stronger language

## Section guidance

### Overview

Keep it plain. State what the project is about and what it tries to answer.

### Problem

State the real business problem, not a generic analytics problem.

### Objective

Write what the analysis is trying to help a team decide.

### Insights

Make each point specific.
Do not make every insight sound equally important.

### Recommendations

Keep them practical.
If the action is narrow, say it clearly.
If the evidence is limited, keep the recommendation proportionate.

## Editing checklist

Before publishing, check:

- Is this sentence more polished than it is specific?
- Am I repeating the same sentence structure from other case studies?
- Does this claim sound stronger than the actual evidence?
- Could this be written in plainer language without losing meaning?
- Does this sound like an analyst, or like generated portfolio copy?

## Final standard

The writing should feel:

- credible
- readable
- commercially aware
- slightly individual

It should not feel:

- inflated
- interchangeable
- over-smoothed
- written by a model trying to sound smart
